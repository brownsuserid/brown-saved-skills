## Overview

Search Google Meet transcripts stored in Google Drive across both AITB and BB organizations. Transcripts are auto-generated by Google Meet and stored as Google Docs. Some meetings also have Gemini AI-generated summary notes.

**Use cases:**
- "What did I discuss with Joel Hans?", search by person name
- "Meetings about cybersecurity", search by topic keyword
- "What happened in the BBI inbox meeting last week?", search by meeting name + date
- "Find action items from the AITB planning meeting", search + extract key points

## Transcript Format

All transcripts follow the same Google Meet auto-generated structure:

```
{Meeting Title}
Date: {Day, Month Date, Year} Time: {Start} - {End} ({Timezone})
Attendees
{Name} ({email})
{Name} ({email})
Transcript with timestamps
##{Speaker Name} {spoken text}
##{Speaker Name} {spoken text}
```

**Gemini notes** (subset of meetings) have a different structure:
```
{Meeting Title}
Summary
{paragraph summary}
Details
{topic sections with (HH:MM:SS) timestamps}
Suggested next steps
{action items with owners}
```

**Naming patterns:**
- Raw transcripts: `{Meeting Title}/Transcript- {YYYY-MM-DD}`
- Gemini notes: `{Meeting Title} - {YYYY/MM/DD HH:MM TZ} - Notes by Gemini`
- BB prefixes: `(BBI)` = internal, `(BBC)` = customer, `BB` = general

## Phase 1: Search Transcripts

Run the search script with the user's query. It searches both drives in parallel.

```bash
python3 ~/.openclaw/skills/maintaining-relationships/scripts/searching-meeting-transcripts/search_transcripts.py \
  --query "search terms" \
  [--account aitb|bb|both] \
  [--max 10]
```

**How search works:**
- Uses `gog drive search` with plain-text full-text search (Google Drive indexes doc content)
- Results are filtered client-side to only include Google Docs in the transcript folders (requests 3x results to account for filtering)
- Searches both accounts by default, or one with `--account`
- Returns: file ID, title, meeting name (parsed), date (parsed from name), account, Drive URL

**Query tips:**
- Person names: `"Joel Hans"`, `"Manny Felix"`
- Topics: `"cybersecurity"`, `"hackathon"`, `"budget"`
- Meeting names: `"BBI inbox"`, `"planning meeting"`
- Combined: `"Joel Hans sponsor"` (narrows to specific conversation topics)
- Content search works, queries match text inside transcripts, not just filenames

---

## Phase 2: Read Transcript

Once search returns matching files, read the full transcript:

```bash
gog docs cat {fileId} --account {account}
```

For multiple matches, read the most recent one first (highest date in filename).

---

## Phase 3: Analyze and Present

After reading the transcript, analyze based on the user's intent:

- **Person lookup**: Extract all statements by/about that person, summarize key topics discussed
- **Topic search**: Find relevant sections, summarize decisions and action items
- **Action items**: Look for commitments ("I'll do X", "let's follow up on Y", next steps)
- **Meeting context**: Provide attendees, date, duration, and topic summary

Present findings structured as:

```
Meeting: {title}
Date: {date}
Attendees: {names}
Account: {AITB or BB}

Summary:
{2-3 sentence summary of what was discussed}

Key Points:
- {point 1}
- {point 2}

Action Items (if any):
- {item}: {owner}
```

If Gemini notes exist for the same meeting, prefer those for summaries (they're pre-analyzed). Use raw transcripts for exact quotes or detailed context.

## Guardrails

- **Read-only**: This skill only reads transcripts, never modifies or deletes Drive files
- **Privacy**: Transcripts may contain sensitive business discussions. Present only what's relevant to the query
- **Account context**: Always report which account (AITB vs BB) a transcript came from so Aaron knows the organizational context
- **Large transcripts**: Some transcripts are 20+ KB. When presenting, summarize rather than dumping the full text unless Aaron asks for verbatim content
- **Duplicates**: Some meetings have duplicate transcript files (auto-save artifacts). Deduplicate by title + date when presenting results
