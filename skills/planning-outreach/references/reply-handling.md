# Reply Handling

How to classify inbound responses and what to do with each type. Always classify responses BEFORE updating the outreach plan or creating new tasks.

---

## Response Classification

| Type | Indicators | Required Action |
|------|-----------|-----------------|
| Interest | Asks questions, requests more info, agrees to a meeting, positive acknowledgement | Advance qualification; propose meeting; compress timing |
| Objection | Pushback on value, timing, relevance, or need | Acknowledge concern; probe root cause; provide evidence |
| Confusion | Unclear why they were contacted; asks for context | Clarify relevance in ≤4 sentences; re-establish value prop |
| Hostile | Angry tone, threatening, demands to stop contact | Do NOT respond; create Human Communication task to owner; archive all outreach tasks |
| Out-of-Office | Auto-reply with return date | Pause all outreach; set ALL task due dates to return date or later |
| Data Provision | Provides email, phone, title, referral, or other info | Update contact/org record; acknowledge; continue sequence |
| Rejection / Opt-Out | "Not interested", "remove me", "unsubscribe", explicit DNC request | Mark deal DISENGAGED; archive ALL outreach tasks; create ZERO new tasks |

---

## Disengagement Rules

| Scenario | Action |
|----------|--------|
| Full sequence completed with no reply | Mark deal Disengaged; log learning in Plan Deviations Log |
| Explicit "not interested" or decline | Respect immediately; archive all tasks; note in Outreach State |
| DNC (Do Not Contact) request | Immediate stop; flag the Contact record (compliance field only — do not update via enrichment); notify owner |
| Hostile response | Immediate stop; escalate to owner via Human Communication task; archive all outreach tasks |

**After any disengagement scenario: create zero additional outreach tasks.** The only permitted task is a Human Communication to the owner.

---

## Out-of-Office Protocol

1. Identify the return date from the auto-reply.
2. Set the status of all in-progress outreach tasks to "On Hold" or adjust due dates.
3. Set ALL new task due dates to the return date or later (never earlier).
4. Note the OOO and return date in the Outreach State section of the plan.
5. Add a Conditional Forward Planning entry for the resumption scenario.

If no return date is provided, default to 2 weeks out and note the assumption.

---

## Objection Handling Patterns

| Objection | Pattern |
|-----------|---------|
| Timing ("not now", "bad timing", "reach out later") | Acknowledge; ask for a better time; set a future task to the stated timeframe; do not push |
| Budget ("too expensive", "no budget") | Reframe value; offer relevant proof points; do not discount in outreach |
| Authority ("not my decision", "need to check with my team") | Ask for referral to decision maker; add new stakeholder to Research task |
| Need ("we don't need this", "already have a solution") | Probe deeper with one open question; share a relevant case study or result |
| Competitor ("we use X", "we're happy with our current provider") | Acknowledge their current solution; differentiate on specific points without disparaging competitor |
| No time ("too busy") | Offer a shorter format (5-minute call, one-paragraph summary); respect if declined again |

**General objection rule:** Never argue. Acknowledge → understand root cause → respond with evidence or a reframe. If the objection is repeated after one follow-up, treat it as a soft rejection.

---

## Positive Response Protocol

When a contact responds positively (interest, questions, meeting agreement):

1. Classify as Interest and record in Outreach State.
2. Compress timing — follow up within 24–48 hours.
3. Advance the Qualification Tracker (update F.L.O.O.R. / B.A.N.T. based on what they shared).
4. Propose a next step (meeting, discovery call, demo) that matches the interest level.
5. Set authority level based on the nature of the next step (meeting scheduling = Level B minimum).
6. Notify the deal owner if this is a significant positive signal.

---

## Multi-Threading on New Stakeholder Discovery

When a response reveals a new stakeholder or decision maker:

1. Create a Research task to gather contact details on the new stakeholder.
2. Add the stakeholder to the Deal's stakeholder map field in Airtable.
3. Do not begin outreach to the new stakeholder until the Research task is complete and a Level C approval is obtained.
4. Note the multi-threading intent in the Conditional Forward Planning section.
